# 《星航棋域》小型 Demo 实现与熟人试玩计划

## 1. 目标定义

本计划的目标不是做完整商业 SLG，而是在较短时间内实现一个可试玩、可展示、可收集反馈的小型 Demo，用来验证以下问题：

1. 飞行棋改造成 SLG 后，玩家是否能快速理解核心玩法。
2. “行动力替代骰子”“航线跳跃”“大地图占点”“舰队拦截”是否有策略乐趣。
3. 熟人试玩时，是否愿意继续玩第二局或主动讨论战术。
4. 这套 Demo 是否足够支撑参赛展示、录屏和后续 PPT 路演。

建议 Demo 控制在 10-15 分钟一局，支持 1 人体验 AI 对手，或 2-4 人本地/线上轻量对战。

## 2. Demo 范围

### 2.1 必做内容

为了保证 Demo 可以按期完成，第一版只做最小可行玩法：

| 模块 | 内容 |
| --- | --- |
| 六边形地图 | 15x15 左右的小型星域地图 |
| 玩家基地 | 4 个出生基地，对应四色阵营 |
| 单位系统 | 每名玩家 3-4 支舰队 |
| 行动力系统 | 移动、占领、攻击消耗行动力 |
| 战争迷雾 | 未探索区域不可见 |
| 资源点 | 占领后每回合/每分钟产出资源 |
| 折跃门 | 占领后可进行远距离跳跃 |
| 战斗结算 | 简化兵种克制和战力计算 |
| 中央星门 | 占领后持续获得胜利积分 |
| 胜负条件 | 规定时间内积分最高，或率先达到指定积分 |

### 2.2 暂不做内容

以下内容适合放到后续版本，不建议第一版开发：

| 暂缓内容 | 原因 |
| --- | --- |
| 完整主城建设 | 开发量较大，Demo 阶段不核心 |
| 复杂科技树 | 会增加数值和平衡成本 |
| 完整联盟系统 | 熟人小范围试玩可先用阵营替代 |
| 大规模多人同屏 | 技术成本高，不适合首版 |
| 复杂 AI NPC | 可先用简单规则 AI 替代 |
| 付费系统 | Demo 和试玩阶段不需要 |

第一版要做的是“好玩的小战场”，不是“完整的大 SLG”。

## 3. 推荐技术路线

### 3.1 首选方案：网页 Demo

推荐使用网页技术实现，原因是部署和分享最方便。熟人试玩只需要打开链接，不需要安装客户端。

推荐技术栈：

| 模块 | 技术 |
| --- | --- |
| 开发语言 | TypeScript |
| 前端框架 | React |
| 构建工具 | Vite |
| 地图渲染 | Canvas 2D 或 PixiJS |
| UI 组件 | 原生 CSS / Tailwind CSS / shadcn-ui 任选其一 |
| 状态管理 | Zustand 或 React Context |
| 数据配置 | JSON 配表 |
| 本地存档 | localStorage |
| 在线部署 | Vercel / Netlify / GitHub Pages |
| 反馈收集 | 飞书表单 / 腾讯文档 / Google Forms |

如果开发者不熟悉 PixiJS，首版可以直接使用 Canvas 2D。六边形地图、单位图标、移动范围、战斗特效都可以先用 Canvas 绘制。

### 3.2 备选方案：Unity Demo

如果团队中有人熟悉 Unity，也可以用 Unity 制作。

适合 Unity 的情况：

- 想要更强的视觉表现。
- 后续可能继续做成独立客户端。
- 团队成员有 Unity 经验。
- 希望快速使用拖拽、粒子、动画和音效。

Unity 技术条件：

| 模块 | 技术 |
| --- | --- |
| 引擎 | Unity 2022 LTS 或 Unity 6 LTS |
| 开发语言 | C# |
| 地图 | Tilemap 或自定义 Hex Grid |
| 数据 | ScriptableObject / JSON |
| 部署 | WebGL 或 Windows 包 |
| 分享 | itch.io / 网盘 / WebGL 链接 |

如果目标是熟人快速试玩，Unity WebGL 可行，但加载速度和移动端兼容性通常不如网页原生方案。

### 3.3 推荐结论

若只有 1-3 人、时间有限，建议使用：

```text
React + TypeScript + Vite + Canvas 2D + JSON 配表 + Vercel 部署
```

这条路线开发门槛低、展示方便、修改速度快，也适合后续把 Demo 截图和录屏放进参赛材料。

## 5. Demo 玩法规则建议

### 5.1 基础规则

Demo 可采用回合制，减少实时同步和操作压力。

建议规则：

- 每局 20 回合。
- 每名玩家每回合恢复 6 点行动力。
- 每支舰队移动 1 格消耗 1 点行动力。
- 进入折跃门后，可消耗 3 点行动力跳到己方控制的另一座折跃门。
- 占领普通资源点消耗 2 点行动力。
- 攻击敌方单位消耗 2 点行动力。
- 中央星门每占领 1 回合获得 10 分。
- 20 回合结束后积分最高者获胜。

回合制更适合小范围试玩，因为玩家可以边玩边讨论，开发也更简单。

### 5.2 单位设计

第一版保留 3 种单位：

| 单位 | 攻击 | 防御 | 移动 | 特性 |
| --- | --- | --- | --- | --- |
| 截击机 | 6 | 3 | 4 | 攻击轰炸艇 +30% |
| 轰炸艇 | 8 | 2 | 3 | 攻击据点和星门 +30% |
| 预警舰 | 2 | 5 | 3 | 周围 2 格视野，友军命中 +10% |

每名玩家初始拥有：

- 2 支截击机
- 1 支轰炸艇
- 1 支预警舰

### 5.3 地图规则

地图建议使用轴向坐标 Hex Grid。

地块类型：

| 地块 | 规则 |
| --- | --- |
| 普通星域 | 常规移动 |
| 资源点 | 占领后每回合 +2 资源 |
| 星云 | 进入后消耗 +1，视野降低 |
| 陨石带 | 防守方防御 +20% |
| 折跃门 | 控制后可跳跃 |
| 中央星门 | 占领后获得胜利积分 |

### 5.4 战斗规则

Demo 战斗不需要复杂公式，建议使用确定性结算加少量随机。

```text
攻击方战力 = 攻击力 * 克制系数 * 地形系数 * 随机系数
防守方战力 = 防御力 * 地形系数 * 支援系数 * 随机系数
```

随机系数控制在 0.9-1.1，避免纯随机影响策略判断。

战斗结果：

- 攻击方战力高于防守方 20% 以上：防守方被击退回基地。
- 双方战力接近：双方各损失 1 点耐久，防守方留在原地。
- 攻击失败：攻击方损失 1 点耐久并停止行动。

## 6. 技术实现模块拆分

### 6.1 工程结构建议

```text
src/
  data/
    map.json
    units.json
    tiles.json
  game/
    hex.ts
    movement.ts
    combat.ts
    vision.ts
    scoring.ts
    ai.ts
  components/
    GameCanvas.tsx
    Hud.tsx
    UnitPanel.tsx
    TurnPanel.tsx
    ResultModal.tsx
  styles/
    game.css
```

### 6.2 核心模块

| 模块 | 功能 |
| --- | --- |
| hex.ts | 六边形坐标、距离、邻居、范围计算 |
| movement.ts | 行动力消耗、可移动范围、路径校验 |
| vision.ts | 战争迷雾、单位视野、雷达视野 |
| combat.ts | 战斗公式、克制、击退和耐久 |
| scoring.ts | 中央星门积分、资源点积分 |
| ai.ts | 简单 AI 行动逻辑 |
| GameCanvas.tsx | 地图和单位绘制 |
| Hud.tsx | 回合、资源、积分、当前玩家 |

### 6.3 简单 AI 逻辑

第一版 AI 不需要很聪明，只要能制造压力。

AI 优先级：

1. 如果能攻击附近低血量敌人，优先攻击。
2. 如果中央星门无人占领，派最近单位前往。
3. 如果附近有未占领资源点，优先占领。
4. 如果有己方折跃门，尝试向中央推进。
5. 否则向最近敌方单位移动。

## 7. 美术与音效条件

### 7.1 美术最低要求

第一版不需要精美立绘，但需要清晰识别。

需要素材：

| 素材 | 数量 | 要求 |
| --- | --- | --- |
| 地块底图 | 5-6 个 | 普通、资源、星云、陨石、折跃门、星门 |
| 单位图标 | 3 个 | 截击机、轰炸艇、预警舰 |
| 阵营颜色 | 4 套 | 红、蓝、绿、黄 |
| UI 图标 | 5-8 个 | 行动力、资源、攻击、占领、跳跃 |
| 背景图 | 1 张 | 星空或星图背景 |

也可以先用几何图形代替：

- 三角形代表截击机
- 菱形代表轰炸艇
- 圆形代表预警舰
- 发光六边形代表折跃门
- 大型环形图标代表中央星门

### 7.2 音效建议

音效不是必需，但会明显提升试玩感受。

建议加入：

- 点击音
- 移动音
- 折跃音
- 战斗爆炸音
- 占领成功音
- 胜利/失败音

可以使用免费音效库，但需要注意版权。

## 8. 熟人试玩准备

### 8.1 试玩方式

推荐两种试玩方式：

| 方式 | 适合场景 |
| --- | --- |
| 线上链接试玩 | 发给朋友自行体验，收集问卷 |
| 语音会议陪玩 | 观察玩家操作，记录卡点 |

第一轮最好采用“语音会议陪玩”，因为你能直接看到玩家哪里不理解、哪里觉得有趣。

### 8.2 试玩人数

建议分三轮：

| 轮次 | 人数 | 目的 |
| --- | --- | --- |
| 第 1 轮 | 2-3 人 | 找明显 Bug 和理解障碍 |
| 第 2 轮 | 5-8 人 | 验证玩法是否有趣 |
| 第 3 轮 | 10-20 人 | 收集稳定反馈和数据 |

不要一开始就发给很多人。第一版一定会有明显问题，小范围先打磨更有效。

### 8.3 试玩前说明

给玩家的说明应控制在 1 分钟内：

```text
这是一个飞行棋灵感的星际 SLG 小 Demo。
你的目标是在 20 回合内探索地图、占领资源点和中央星门。
移动和攻击都消耗行动力，占领中央星门会持续得分。
最后积分最高的阵营获胜。
```

如果玩家需要读很长说明才能玩，说明新手引导还不够好。

### 8.4 试玩问卷

建议问卷不超过 10 题：

1. 你是否能在 3 分钟内理解目标？
2. 你是否能理解行动力的作用？
3. 航线/折跃门是否让你觉得有策略价值？
4. 战斗结果是否容易理解？
5. 你觉得最有趣的部分是什么？
6. 你觉得最困惑的部分是什么？
7. 你是否愿意再玩一局？
8. 如果加入多人对战，你是否愿意和朋友一起玩？
9. 你觉得画面信息是否清楚？
10. 请给当前 Demo 打 1-10 分。

### 8.5 需要记录的数据

除了主观问卷，最好记录一些行为数据：

| 数据 | 用途 |
| --- | --- |
| 单局时长 | 判断节奏是否过慢 |
| 玩家第几回合首次战斗 | 判断冲突出现是否太晚 |
| 玩家是否使用折跃门 | 判断核心机制是否被看见 |
| 中央星门占领次数 | 判断终局目标是否有效 |
| 玩家失败原因 | 判断规则是否不公平或不清楚 |

第一版可以手动记录，不一定要接入埋点系统。

## 9. 开发里程碑

### 第 1 周：规则和原型

目标：可以在本地跑起来，看到地图和单位。

任务：

- 确定 15x15 地图布局。
- 完成单位和地块数值表。
- 搭建 React + Vite 项目。
- 实现六边形地图绘制。
- 实现单位选中和移动范围显示。

验收标准：

- 可以打开网页看到完整地图。
- 可以选中单位并移动。
- 移动会消耗行动力。

### 第 2 周：核心玩法

目标：形成可玩的单机闭环。

任务：

- 实现资源点占领。
- 实现中央星门积分。
- 实现折跃门跳跃。
- 实现战争迷雾。
- 实现基础战斗结算。
- 实现胜负结算界面。

验收标准：

- 可以完整玩完一局。
- 有明确胜负。
- 玩家能感受到占点、跳跃和战斗。

### 第 3 周：试玩版本

目标：可以发给熟人试玩。

任务：

- 加入简单 AI。
- 优化 UI 信息显示。
- 加入基础音效和动效。
- 修复主要 Bug。
- 部署到 Vercel 或 Netlify。
- 准备试玩问卷。

验收标准：

- 外部玩家打开链接即可游玩。
- 不需要开发者手把手解释也能完成一局。
- 能收集反馈。

### 第 4 周：迭代和参赛包装

目标：根据试玩反馈优化，并输出参赛展示素材。

任务：

- 根据反馈调整数值和地图。
- 录制 2-3 分钟 Demo 视频。
- 截取关键玩法截图。
- 整理试玩数据和玩家反馈。
- 更新参赛 PPT。

验收标准：

- Demo 运行稳定。
- 核心玩法能在视频中被看懂。
- PPT、文档、录屏能形成完整提交材料。

## 11. 风险与应对

| 风险 | 表现 | 应对 |
| --- | --- | --- |
| 玩法太复杂 | 玩家不知道该干什么 | 第一版减少系统，只保留移动、占点、战斗、跳跃 |
| 节奏太慢 | 前几回合没有冲突 | 缩小地图，让中央目标更早出现 |
| 折跃门存在感弱 | 玩家不用核心机制 | 把资源点和中央星门放在折跃门附近 |
| 战斗看不懂 | 玩家觉得输赢随机 | 显示战力对比、克制提示和地形加成 |
| 美术太粗糙 | 第一印象差 | 保证颜色、图标和信息层级清楚 |
| 多人同步困难 | 联机 Bug 多 | 第一版先做单机热座或 AI 对战 |

## 12. 第一版成功标准

Demo 第一版不需要完美，但应达到以下标准：

1. 玩家能在 3 分钟内理解目标。
2. 玩家能在 10-15 分钟内完成一局。
3. 玩家至少主动使用一次折跃门。
4. 玩家至少发生一次战斗。
5. 玩家能说出“我下一局想换一种打法”。
6. 试玩问卷平均分达到 7 分以上。
7. Demo 可以录制出清晰的参赛展示视频。

## 13. 推荐执行顺序

最推荐的执行顺序如下：

1. 先确定地图和规则，不急着做美术。
2. 用简单几何图形做可玩原型。
3. 只实现单机或热座，不先做复杂联网。
4. 找 2-3 个熟人边玩边观察。
5. 根据卡点调整 UI 和规则。
6. 再做基础美术、音效和动效。
7. 部署线上链接，扩大到 10-20 人试玩。
8. 收集反馈后整理成参赛材料。

## 14. 总结

《星航棋域》的小型 Demo 应优先验证“飞行棋灵感是否能自然转化为 SLG 策略乐趣”。第一版不需要完整商业系统，也不需要复杂联机。最有效的方案是使用网页技术制作一个轻量、直观、可快速分享的回合制战术沙盘。

只要 Demo 能让玩家完成“探索、移动、占点、跳跃、战斗、争夺中央星门”的完整体验，就已经足以支撑小范围试玩和参赛展示。后续再根据反馈逐步加入主城建设、科技树、联盟系统和更丰富的大地图事件。
